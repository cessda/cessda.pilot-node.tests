# EOSC Pilot Node Dashboard

A browser-based monitoring dashboard for EOSC Beyond Pilot Nodes. It reads JSON reports generated by three Java data-collection checks and presents capability, catalogue, and uptime data across all registered nodes. Reports can also be triggered directly from the UI.

---

## File Naming Conventions

| File | Description |
| ---------------------------------- | -------------------------------------------------------------------- |
| `node_registry_summary.json`       | Central registry — fixed name, always at the `data/` root           |
| `endpoint_report.json`             | Endpoint capability report (generated by `CheckNodeCapabilities`)    |
| `catalogue_services_report.json`   | Catalogue services report (generated by `CheckCatalogueServices`)    |
| `argo_uptime_report.json`          | ARGO uptime report (generated by `CheckServiceUptime`)               |

Node folder names must exactly match the `node_name` field in `node_registry_summary.json`.

---

## Architecture

```
data/
├── node_registry_summary.json        ← registry of all nodes
├── CESSDA/
│   ├── endpoint_report.json
│   ├── catalogue_services_report.json
│   └── argo_uptime_report.json
├── CLARIN/
│   └── ...
└── ...
```

The Spring Boot backend serves the dashboard files and exposes a REST API at `/api/`. The dashboard HTML files make JSON fetches relative to that path and require an HTTP server — opening them directly via `file://` will not work because browsers block local file fetches.

## Data Collection Checks

Three Java classes generate the report files. They can be run from the command line or triggered via the dashboard UI.

### CheckNodeCapabilities

Queries the EOSC Node Registry and writes `node_registry_summary.json` and per-node `endpoint_report.json` files. Requires a node API key configured in `application.properties`.

```
CheckNodeCapabilities
```

Triggered via: `POST /api/run/node-capabilities`

### CheckCatalogueServices

Queries a node's Resource Catalogue API, checks the availability of each listed service's webpage, and writes `catalogue_services_report.json`.

```
CheckCatalogueServices NODE_NAME [api_base_url] [quantity] [dashboard_dir]
```

| Argument        | Required | Default                             | Description |
| --------------- | -------- | ----------------------------------- | ----------- |
| `NODE_NAME`     | Yes      | —                                   | Node name; must match `node_registry_summary.json` |
| `api_base_url`  | No       | CESSDA staging URL                  | The node's Resource Catalogue API base URL — taken from the `Resource Catalogue` capability endpoint in `endpoint_report.json` with `/api/service/all` appended |
| `quantity`      | No       | `10`                                | Maximum number of services to retrieve |
| `dashboard_dir` | No       | `../dashboard/data`                 | Output root directory |

Triggered via: `POST /api/run/catalogue-services` with body `{ "node": "...", "catalogueUrl": "..." }`

### CheckServiceUptime

Queries the ARGO Service Monitoring API and writes `argo_uptime_report.json` covering the last 30 days. Requires an ARGO API key.

```
CheckServiceUptime NODE_NAME API_KEY [START_DATE] [END_DATE] [dashboard_dir]
```

| Argument        | Required | Default          | Description |
| --------------- | -------- | ---------------- | ----------- |
| `NODE_NAME`     | Yes      | —                | Node name; must match `node_registry_summary.json` |
| `API_KEY`       | Yes      | —                | ARGO API key — entered via the dashboard modal or supplied on the command line; never persisted |
| `START_DATE`    | No       | 30 days ago      | `YYYY-MM-DD` |
| `END_DATE`      | No       | Today            | `YYYY-MM-DD` |
| `dashboard_dir` | No       | `../dashboard/data` | Output root directory |

Triggered via: `POST /api/run/service-uptime` with body `{ "node": "...", "apiKey": "..." }`

## Adding a New Node

When a new node has been added to the [EOSC Node Registry](https://node-devel.eosc.grnet.gr/federation/eosc-beyond/home), run `CheckNodeCapabilities`. This updates `node_registry_summary.json` and creates `data/NEW_NODE_NAME/endpoint_report.json`. The new node will then appear on the dashboard automatically.

To populate the remaining reports, run `CheckCatalogueServices` and `CheckServiceUptime` against `NEW_NODE_NAME`. The generated files are placed in `data/NEW_NODE_NAME/`.

Missing reports are handled gracefully throughout the UI:

- The landing page shows "No report" rows and greyed-out chips on the node card.
- The node detail page shows "Not available" panels for each missing report, rather than errors.
- If all three reports are absent for a node, the detail page shows: _"No report files found for NODE"_.

## MVP Requirement

A node is considered to meet the **Minimum Viable Product (MVP)** requirement if its `endpoint_report.json` contains both of the following capabilities with `status: "Available"`:

- `AAI`
- `Resource Catalogue`

The MVP status is shown:

- On the **landing page** — as an `X/Y Meet MVP` tile in the summary strip, and as a row on each node card.
- On the **node detail page** — as a ✓ or ✗ card in the overview strip.

## Running Locally

The dashboard requires a running Spring Boot backend for the `/api/` routes. For static file development only (no run-checks functionality), any HTTP server will do.

```bash
# Spring Boot (full functionality)
mvn spring-boot:run
# or
java -jar target/pilot-node-dashboard-1.0.0-SNAPSHOT.jar

# Static file server (read-only, no checks)
python3 -m http.server 8080
# or
npx serve .
```

Then open `http://localhost:8080/index.html`.

## Configuration

Set the following properties in `application.properties`:

| Property              | Description |
| --------------------- | ----------- |
| `dashboard.data-dir`  | Path to the data directory (e.g. `../dashboard/data`) |
| `check.node-name`     | Default node name for `CheckNodeCapabilities` |
| `check.api-key-node`  | API key for `CheckNodeCapabilities` |
| `check.api-key-argo`  | Fallback ARGO API key for `CheckServiceUptime` (the dashboard modal takes precedence) |

## Navigation

### Landing page — `index.html`

Shows all registered nodes in a card grid. Each card displays:

- Overall health badge (Healthy / Degraded / Down / No data)
- Endpoint availability (available / total, with percentage)
- Catalogue services (active / total)
- ARGO uptime percentage
- MVP status
- Report availability chips (present = blue, missing = grey)

The summary strip at the top shows aggregate totals across all nodes: total nodes, healthy nodes, degraded or missing, capability uptime percentage, available capabilities, and nodes meeting the MVP requirement.

Click any node card to open its detail page.

### Node detail page — `node.html#NODENAME`

The node name is carried in the URL hash fragment (e.g. `node.html#CESSDA`), which means navigation works correctly with server-side URL rewriting and the page can be bookmarked or shared directly.

The page shows:

- **Overview strip** — endpoints available, active catalogue services, average ARGO uptime, and MVP status at a glance.
- **Endpoint Connectivity Report** — one card per capability with type, URL, HTTP status, and version.
- **Catalogue Services Report** — table of services with name, abbreviation, URL, and availability status.
- **ARGO Uptime Report** — per-service uptime bars with percentage, days monitored, and reporting period.

Use the **Switch node** dropdown (top right) to move directly to another node's detail page without returning to the landing page.

The **Run checks** menu (top right) triggers node-specific data collection:

- **Catalogue Services** — runs immediately using the node's Resource Catalogue endpoint; no credentials required.
- **Service Uptime** — prompts for an ARGO API key via a modal. The key is used only for that single request and is never saved or stored anywhere.

The breadcrumb at the top of the page returns to the landing page.

### API Key Security

The ARGO API key entered in the Service Uptime modal is:

- Never written to `localStorage`, `sessionStorage`, or any other persistent storage.
- Cleared from the input field immediately when the modal is closed or confirmed.
- Passed in the HTTP request body to the backend and goes out of scope as soon as the request completes.
- The input field is always blank when the modal is next opened.

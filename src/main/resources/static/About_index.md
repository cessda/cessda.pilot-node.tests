# EOSC Beyond Pilot Node Dashboard

## Overview

The Pilot Node Dashboard gives an at-a-glance view of all federated
pilot nodes registered with the EOSC Beyond infrastructure. It shows
which nodes are reachable, how many catalogue services they expose,
their ARGO uptime, and the compliance tier they have achieved.

The dashboard is a static HTML page. It loads all data from JSON
report files served by a local API and does not contact any external
service at display time.

## Summary Statistics

The bar at the top of the page shows network-wide totals derived
from the most recently collected reports.

- **Total nodes** — the number of nodes in the registry.
- **Healthy** — nodes whose endpoint or ARGO uptime is at or above
  90 %.
- **Degraded / no data** — nodes below that threshold or for which
  no report was found.
- **Capability uptime** — the percentage of all reported endpoint
  capabilities that returned an Available status.
- **Available caps** — the raw available / total count of all
  capabilities across all nodes.
- **Tier 1 · MVP** — nodes that have both AAI and Resource Catalogue
  available.
- **Tier 2 · Standard** — nodes that additionally have Helpdesk and
  Service Monitoring available.
- **Tier 3 · Advanced** — nodes that additionally have all four
  advanced capabilities available.

## Node Cards

Each card represents one registered pilot node. The card shows:

- The node name and its base endpoint URL.
- A status badge (Healthy / Degraded / Down) derived from the
  endpoint or ARGO availability percentage.
- Three report rows summarising endpoint connectivity, catalogue
  services, and ARGO uptime.
- A compliance row showing the highest tier the node has reached.
- Availability chips indicating which of the three report files
  were found for this node.

Clicking a card opens the Node Detail page for that node.

## Compliance Tiers

Federation compliance is measured in three tiers. A node must fully
satisfy each tier before the next one is counted.

| Tier | Label    | Required capabilities                                                                                  |
|------|----------|--------------------------------------------------------------------------------------------------------|
| 1    | MVP      | AAI, Resource Catalogue                                                                                |
| 2    | Standard | Helpdesk, Monitoring                                                                                   |
| 3    | Advanced | Service Accounting, Research Product Accounting, Order Management, Application Deployment Management   |

A node shown as Tier 2 · Standard has satisfied both Tier 1 and
Tier 2. A node that has not satisfied Tier 1 is shown as Below MVP.

## Data Sources

The dashboard reads three types of JSON report file. All files are
served from `/api/data/`.

### Node registry

```text
/api/data/node_registry_summary.json
```

This file lists all registered pilot nodes and records the timestamp
at which the registry was last generated. It is the root from which
all other data is discovered.

### Endpoint connectivity reports

```text
/api/data/{node_name}/endpoint_report.json
```

Generated by the `CheckNodeCapabilities` check. Contains the list of
capability types declared by the node, the endpoint URL tested for
each, the HTTP response code returned, and whether the capability
was Available or not.

### Catalogue services reports

```text
/api/data/{node_name}/catalogue_services_report.json
```

Generated by the `CheckCatalogueServices` check. Contains the
services published in the node's Resource Catalogue, with their
names, URLs, and status values.

### ARGO uptime reports

```text
/api/data/{node_name}/argo_uptime_report.json
```

Generated by the `CheckServiceUptime` check. Contains uptime
percentages for each monitored service endpoint, sourced from the
ARGO monitoring infrastructure.

## Running Checks

The **Run checks** menu in the top-right corner triggers on-demand
data collection for the whole network. The available check is:

- **Node Capabilities** — runs `CheckNodeCapabilities` against every
  registered node and refreshes `endpoint_report.json` for each.

The page must be served from an HTTP server with an active backend
for the Run checks menu to work; it will not function when the file
is opened directly from disk.
